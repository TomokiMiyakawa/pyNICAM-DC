pyNICAM-DC readability refactor -- plan
=======================================
branch: readability   (base: main @ 8dc8077)
status: in progress. This file records the agreed direction; it is not final.


0. GOAL
-------
Restore human readability of the dynamical core WITHOUT losing the GPU
performance stack, and leave a structure that physics + ocean coupling can be
added to without compounding the mess.

Non-goal: re-tuning performance. Behaviour is preserved throughout.

Reading the core means stepping over a fork between "the algorithm" and "the
fast variant" every few dozen lines. Renaming and comments cannot fix this;
only separation can.


2. THE SEPARATION AXIS
----------------------
NOT numpy vs jax. Separating those would duplicate the science (every
algorithm twice, kept in sync forever, two answers to "what does the model
do") and make numpy/jax bit-exactness a permanent chore.

The axis is: does the code change STRUCTURE?

  (1) xp-clean            same maths, one source, runs on numpy and jax
                          via `xp`. THIS IS THE TARGET STATE. Keep.

  (2) backend-incompatible primitives
                          `.at[]` (jax) vs `a[i] = x` (numpy). 157 sites.
                          NOT acceleration -- just incompatibility.
                          FIX BY ABSTRACTION, not separation:
                             a = bk.set_at(a, idx, val)
                          (numpy: in-place then return; jax: a.at[idx].set(val))
                          This alone removes much of the 40 bk.type branches.

  (3) true acceleration   lax.scan / lax.fori_loop / kernel fusion / device
                          residency. Cannot be expressed with `xp`, and is
                          jax-only by nature. SEPARATE THIS.

jit is NOT the boundary. bk.maybe_jit is transparent (numpy: no-op; jax: jit)
and does not change structure, so per-kernel jit belongs on the xp side. A
maybe_jit'd kernel called from inside an outer jit/scan is inlined by JAX and
XLA fuses across it -- so the SAME kernels serve both routes.


3. WHERE THE ROUTES DIVERGE
---------------------------
At the loop drivers, not at the kernels.

  driver time loop      <- divergence 1:  python for  |  lax.scan
    dynamics_step
      nl-loop (large)   <- divergence 2:  python for  |  lax.scan
        kernel calls       [SHARED: xp + maybe_jit]
        vi (small/ns)   <- divergence 3:  python for  |  fori_loop

Pattern: write the loop body ONCE as a pure state -> state function; the
divergence is a ~3-line shim per loop level.

    def _nl_body(state):        # shared, xp, pure
        ...
        return state

    if fuse:
        state = lax.scan(lambda s, _: (_nl_body(s), None), state, length=N)[0]
    else:
        for _ in range(N):
            state = _nl_body(state)

Requires: pure bodies, no bk.type inside the body, updates via the set_at
abstraction, no data-dependent python control flow in the body.

Device residency is a CONSEQUENCE of fusion, not a separate axis: inside a
fused scan everything stays in one XLA program. The RESIDENT_* gates exist to
get residency WITHOUT fusion -> they largely lose their reason to exist.


4. THE GATES
------------
The GPU campaign is essentially finished; the winners are known. ONE axis must
survive because it will be needed again when physics is added:

  KEEP  fusion scope: fuse across the time loop  |  fuse only up to the ns-loop
        This is really the choice "how much of the physics must live inside
        jax": whole-timeloop fusion requires everything inside (incl. physics)
        to be jax-traceable; stopping at the ns-loop lets non-jax-pure physics
        run per step outside the fused region.
  KEEP  backend (numpy/jax) + precision (fp32/fp64)   -- mod_backend already
  KEEP  diagnostics: PROFILE / DTYPE_AUDIT / TIMELOOP_DUMP

  DROP  everything else: promote the winner to default and delete the gate and
        the loser path (RESIDENT_*, HDIFF_HOIST, HDIFF_C2W_CACHE,
        VP0_HOIST_SCRATCH, SINGLE_DRAIN, FORCING_JIT/DEVICE, ONDEVICE_COMM, ...)

Target: 118 -> ~4-5.
Encouraging: _fuse_nlbody and _fuse_nlscan are already the same value
(bk.resident()), i.e. the fine-grained gates are partly collapsed already.


5. THE WARM-UP STATE MACHINE (the root knot)
--------------------------------------------
Today there are THREE routes -- eager / per-nl jit / whole-loop scan -- and the
code TRANSITIONS BETWEEN THEM AT RUNTIME:

  _fuse_warm_calls counts eager nl passes
  _nlbody_steady   = (self._prepost_jit is not None and ...)
  _step_use_scan   = latched at nl==0 once steady
  under scan: the whole nl-loop runs at nl==0, the remaining iterations
              `continue`, and nl==last re-enters for the tracer tail

Why it exists -- the code says so:
  "Run eager until STEADY -- this builds the sub-jits AND the FUSE_PREPOST
   _prepost_jit (else their BUILD would fire inside the _nl_body/scan trace
   = illegal)"

i.e. the sub-jits are built LAZILY, and you cannot build a jit inside a trace.
Everything else (warm counter, steady test, latch) is a consequence.

Build them at setup and the whole cascade disappears.

Feasibility CONFIRMED: _prepost_fn closes over msc, bndc, GSGAM2, pre_bs,
rho_bs, cnst scalars and index constants only -- all run-constant, all known
once dynamics_setup has run (the code's own comment certifies them
"run-constant ... verified set-once"). Only 4 array args are traced.
bk.device_consts already exists as the mechanism for device-resident
run-constants.

Note the difference between the two lazily built jits:
  _prepost_jit                 closes run-constants only -> buildable at setup TODAY
  _nl_body_jit / _nl_scan_jit  close over 17 per-step nonlocals; the cached jit
                               is bound to the FIRST dynamics_step's closure and
                               reused for every later step. That is only correct
                               if all 17 are run-constant or write-only -- an
                               invariant nobody can see or check. Purifying the
                               body is therefore CORRECTNESS INSURANCE, not
                               cosmetics.


6. JIT POLICY
-------------
  [+] one mechanism: bk.maybe_jit. Raw jax.jit is retired.
      (today: maybe_jit 51 / raw jax.jit 55 -- two ways to do one thing)
  [+] scope: hot path only -- time-loop, shape-stable, pure kernels.
      NOT setup / IO / diagnostics / one-shot code.
      This is ALREADY true in practice: maybe_jit appears in mod_numfilter,
      mod_vi, mod_dynamics, mod_src, mod_src_tracer, mod_cnvvar, mod_bndcnd,
      mod_oprt, mod_thrmdyn -- and in NO setup module (grd/gmtr/vmtr/adm).
      So: formalize the existing scope, do not widen it.
  [+] static_argnames only for genuinely static, low-cardinality arguments.


7. MEASUREMENT
--------------
Current practice: compile lands in the first ~2 steps, then a warm-up follows;
those are discarded and the steady sec/step is measured after.

That practice stays valid. But note today's design ENTANGLES compile with
warm-up: the compile firing point is decided by the warm-up state machine, and
`_fuse_warm_calls >= num_of_iteration_lstep` means it MOVES with the RK order
(RK2=2 / RK3=3 / RK4=4). The discard window therefore cannot be derived, only
guessed.

  after (5)   compile is confined to the first call     -> clean phase boundary
  with AOT    jax.jit(f).lower(ShapeDtypeStruct...).compile() at setup
              -> compile becomes a NAMED, directly measured setup phase;
                 the loop is pure hardware warm-up + steady; no subtraction.
              (shapes are static after setup, so no real data is needed)

  verify, don't assume:  JAX_LOG_COMPILES=1 logs every compile.
  Make "zero compiles inside the measurement window" a regression test --
  it also catches retraces introduced later by the set_at abstraction or by
  new physics.

Note: jax.jit(f) does NOT compile; it wraps. Compilation fires on first call.
Moving CONSTRUCTION to setup does not by itself move COMPILATION.

10. PHYSICS CATEGORIES AND FILE LAYOUT
--------------------------------------
Physics enters NICAM-DC and full NICAM through structurally different doors,
so they must stay separate categories:

  DC style     forcing is ADDED TO f_TEND inside the dynamics step
               (mod_forcing_driver, one thin AF_TYPE hook)
  NICAM style  physics is OPERATOR-SPLIT: physics_step modifies the prognostic
               variables in place between dynamics steps -- CP->SC->MP->RD->
               SFC/TB in fixed order with sub-stepping -- using
               prgvar / diagvar / sfcvar

Measured layout of the two Fortran trees:

  nicamdc  src/nhm/forcing/      mod_af_dcmip, mod_af_heldsuarez,
                                 mod_af_trcadv, mod_forcing_driver, mod_nudge
           src/nhm/share/dcmip/  kessler, simple_physics_v5_1/v6, Terminator,
                                 dcmip_initial_conditions_*, supercell_test,
                                 tropical_cyclone_test
           (no physics/ at all)

  nicam    nhm/physics/          mod_af_driver -> mod_af_heldsuarez,
                                     mod_af_forcing, mod_af_gcss, mod_af_twpice
                                 mod_fr_driver -> mod_fr_heldsuarez,
                                     mod_fr_rigidrot        (selected by FR_TYPE)
                                 mod_movingframe
                                 <- artificial forcing lives INSIDE physics,
                                    behind TWO parallel dispatchers
           nhm/share/            mod_nudge, mod_frcvar
           (no forcing/ at all)

So "forcing vs physics" is a DC-ism: full NICAM regards artificial forcing as
a kind of physics and files it under physics/.

The prefixes encode SCOPE, not lineage:
    af_   a complete artificial forcing   (BOTH trees use it)
    fr_   a friction term only            (nicam only, via FR_TYPE/mod_fr_driver)
Both trees have mod_af_heldsuarez; nicam ALSO carries mod_fr_heldsuarez, a
friction-only fragment reached through a second dispatcher. So af_ does not
mark a lineage, and mirroring both trees verbatim WOULD leave forcing-like
code in two directories, which a reader who does not know the history cannot
disentangle. That confusion outweighs a literal 1:1 directory mapping.

DECISION: keep the name nhm/forcing/, but make it a RULE about content rather
than a marker of lineage.

   nhm/forcing/      ALL artificial / idealized forcing, whatever its lineage.
                     Today: af_dcmip, af_heldsuarez, af_trcadv, mod_forcing.
                     If a NICAM-lineage forcing (af_gcss, af_twpice,
                     fr_rigidrot, movingframe) is ever ported it lands HERE,
                     not in physics/.
   nhm/share/dcmip/  self-contained idealized TEST PHYSICS + initial
                     conditions: kessler, simple_physics, terminator, dcmip_ic
                     (matches nicamdc src/nhm/share/dcmip/)
   nhm/physics/      real parameterizations only: CP / MP / RD / TB / SF
                     (nicam NICAM/nhm/physics/ minus its af_*/fr_*)

   One concept -> one place. "Where is the forcing?" has exactly one answer,
   and so does "where is the physics?".

   Cost: a documented deviation from nicam for its forcing modules
   (af_driver/af_forcing/af_gcss/af_twpice, fr_driver/fr_heldsuarez/
   fr_rigidrot, movingframe). In practice most will never be ported:

     - DC's mod_af_heldsuarez is a SUPERSET of nicam's. Verified by diff:
       identical constants (sigma_b 0.7, k_f 1/1day, k_a 1/40day, k_s 1/4day,
       DT_y 60 K, Dth_z 10 K) and an identical T_eq formula, but DC's adds the
       moist-HS variant (Thatcher & Jablonowski 2016: T_eq0 294, DT_y 65) and
       uses the configurable RP precision where nicam hardcodes real(8).
       The already-ported DC version therefore covers nicam's.
     - nicam's mod_fr_heldsuarez is NOT a second copy of the same physics: it
       is the friction term ALONE (no k_a/k_s, no T_eq) -- a fragment of
       nicam's own FR_TYPE decomposition, redundant once the complete af_
       version is present.

   so the deviation is largely notional, while the rule makes the confusion
   structurally impossible.

   Rejected alternatives:
     rename to dc_forcing/   marks the lineage but still leaves forcing in two
                             places (and a hyphen is illegal in a package name)
     move forcing/ into share/dcmip/
                             mislabels Held-Suarez, which is Held & Suarez
                             1994, not a DCMIP case

DRIFT TO FIX: kessler.py, simple_physics.py and terminator.py sit in
nhm/forcing/, but nicamdc keeps them in share/dcmip/, and mod_dcmip_ic.py is
already in nhm/share/. The DCMIP set is split across two places. Moving them
to nhm/share/dcmip/ realigns with the Fortran AND implements the rule above.

All three categories coexist permanently: the idealized tests never go away.

The category boundary IS the fusion-scope boundary: physics_step sits inside
the driver's time loop, so fusing across the time loop requires physics_step
to be jax-traceable, whereas stopping at the ns-loop lets non-traceable
physics run outside the fused region.

The physics scheme STYLE is already correct and is the template to copy:
kessler.py and simple_physics.py are module-level, xp-parametrized (xp=np,
numpy-first), column-vectorized functions that return new arrays. kessler's
rain sedimentation has a DATA-DEPENDENT trip count (rainsplit.max() from CFL),
illegal under jit -- and it solves it exactly as section 3 prescribes: the
body (_rain_body) is shared, and only the loop driver branches
(python for | lax.fori_loop with a traced bound). mod_dynamics is the module
that deviates from this pattern, not the physics.

  Known gap: simple_physics TC_PBL_mod (Bryan PBL) is not xp-wired -- it has a
  downward recurrence and raises on the jax backend. Same fix as kessler:
  shared body, branch only at the loop driver.

Open: when mod_nudge is ported, nicamdc files it under forcing/ and nicam
under share/ -- pick one.
