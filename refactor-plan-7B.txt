2. TARGET DESIGN
----------------
Define one registered pytree, e.g. (names illustrative):

    import jax  # or numpy-safe: a plain NamedTuple works as a pytree under jax and a
                # tuple under numpy; register a dataclass with jax.tree_util if fields grow.
    class State(NamedTuple):        # a pytree: jax threads it through scan; numpy = a tuple
        prog:     Array            # PROG          [...,0:6]
        prog_pl:  Array
        diag:     Array            # DIAG          [...,I_*]
        diag_pl:  Array
        progq:    Array            # PROGq         [...,tracers]
        progq_pl: Array
        prog0:    Array            # per-ndyn RK base snapshot
        prog0_pl: Array

The body becomes `state = _nl_body(nl, state)` and the scan carry IS `state`. The
10-tuple return keeps its 4-element FEED (pm/frhog) SEPARATE from the state (feed is
scan ys, not carry). The cross-step form (self._prgvar_d) is `concat(state.prog,
state.progq)`; keep a `State.to_prgvar()` / `from_prgvar()` (or free fns) for the
marshal.

KEY ENABLER: once _nl_body/_nl_body_scan take `state` as an ARG (not closure-capture),
they can be defined at INSTANCE/MODULE level (or built in finalize) -> `_nl_scan_jit =
maybe_jit(scan-over-_nl_body_scan)` moves to dynamics_setup_finalize (the stage-4 tail).
Everything ELSE the body needs from the enclosing scope (the ~40 run-const aliases:
kmin, I_RHOG, rho_bs, large_step_dt, the kernels) is msc/config -> pass `msc` (or a
small frozen Consts) as the OTHER arg. bk/xp already come off msc.

Do it INSIDE-OUT so each step is small and reversible. Suggested order:

  7B-1  Introduce `State` as a THIN alias over the existing 8-tuple: replace the 8
        positional carry slots with one `State` namedtuple at the 5 structural sites
        (_nl_body sig @1283, its carry-return slots @2076, _nl_body_scan unpack/repack
        @2084/2110, _scan_init @2141, _final unpack @2157). Body still reads
        state.prog etc. NO other logic change. On numpy this is a pure rename (tuple ->
        namedtuple) -> EXACTLY 0.0; on resident the scan carry pytree changes shape but
        the values are identical -> EXACTLY 0.0 (namedtuple is a registered pytree).
        VERIFY numpy A/B (0.0) + resident A/B (0.0).                        risk: low

  7B-2  Thread `msc` (or a frozen Consts) into _nl_body as an explicit arg; replace the
        remaining closure-captured run-const aliases inside the body with reads off it.
        Still nested; still 0.0 both backends (pure arg-plumbing).           risk: low-med

  7B-3  Lift _nl_body / _nl_body_scan OUT of dynamics_step (to instance methods taking
        (self, nl, state, msc) / (self, carry, nl, msc)). Build `_nl_scan_jit` in
        dynamics_setup_finalize. Retire the lazy `if _nl_scan_jit is None` build at
        step 0. THIS is the stage-4 tail. On resident this MOVES the build to setup ->
        the residual is the jit-vs-eager FP FLOOR at step 0 (like 2a/4e), NOT 0.0 --
        verify with the PYNICAM_PROVE_FLOOR technique.                        risk: med

  7B-4  Unify the cross-step stash (self._prgvar_d) + the marshal in/out + _step_core
        around State (to_prgvar/from_prgvar). Collapse the self.* host-buffer aliases
        that become redundant. Shrinks dynamics_step.                         risk: med

  7B-5  Fold the msc.dyn.* kernel reads (bndcnd) to take the state explicitly (optional;
        small).                                                               risk: low

Stop after any step -- each is a valid, verified commit. 7B-1/7B-2 alone (name the
state, thread msc) are the cheap, safe wins; 7B-3 is the payoff; 7B-4 the shrink.
