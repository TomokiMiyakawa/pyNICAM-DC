2. THE SEPARATION AXIS
----------------------
NOT numpy vs jax. Separating those would duplicate the science (every algorithm
twice, kept in sync forever, two answers to "what does the model do") and make
numpy/jax bit-exactness a permanent chore.

The axis is: does the code change STRUCTURE?

  (1) xp-clean            same maths, one source, runs on numpy and jax via
                          `xp`. THIS IS THE TARGET STATE. Keep.

  (2) backend-incompatible primitives
                          `.at[]` (jax, immutable) vs `a[i] = x` (numpy). 157
                          sites. NOT acceleration -- just incompatibility.
                          FIX BY ABSTRACTION, not separation:
                             a = bk.set_at(a, idx, val)
                          (numpy: in-place then return; jax: a.at[idx].set(val))

  (3) true acceleration   lax.scan / lax.fori_loop / kernel fusion / device
                          residency. Cannot be expressed with `xp`, jax-only by
                          nature. SEPARATE THIS.

The xp route's CONTRACT is the Python Array API Standard (data-apis.org/
array-api). numpy 2.2.5 and jax 0.6.0 already implement its entry protocol
(x.__array_namespace__() -> numpy / jax.numpy), so it costs no dependency, and
array-api-strict can act as a TEST backend that turns "keep xp close to numpy"
into a tier1 check instead of an eyeball one. Apply strict to the pure kernels,
not globally (comm/oprt use restricted fancy indexing the standard omits).

What the standard does NOT cover -> stays as bk.* helpers:
  - in-place update (bk.set_at). Verified: jax rejects a[i]=x, numpy has no .at[]
  - jit / scan / vmap / residency -- correctly out of scope; that IS route (3)
  - fancy/advanced indexing -- deliberately restricted (comm, oprt)
  - RNG -- a non-issue: the model uses no random numbers at all (measured)

jit is NOT the boundary. bk.maybe_jit is transparent (numpy: no-op; jax: jit),
does not change structure -> per-kernel jit belongs on the xp side. A maybe_jit'd
kernel called inside an outer jit/scan is inlined and XLA fuses across it, so the
SAME kernels serve both routes. (Verified this session on CPU: an inner jit
called under an outer jit is inlined and recompiled as one fused program; pre-
compiling the inner jit standalone is wasted -- only the OUTER compile matters.)


3. WHERE THE ROUTES DIVERGE
---------------------------
At the loop drivers, not at the kernels.

  driver time loop      <- divergence 1:  python for  |  lax.scan
    dynamics_step
      nl-loop (large)   <- divergence 2:  python for  |  lax.scan
        kernel calls       [SHARED: xp + maybe_jit]
        vi (small/ns)   <- divergence 3:  python for  |  fori_loop

Pattern: write the loop body ONCE as a pure state -> state function; the
divergence is a ~3-line shim per loop level.

    def _nl_body(state):        # shared, xp, pure
        ...
        return state

    if fuse:
        state = lax.scan(lambda s,_: (_nl_body(s), None), state, length=N)[0]
    else:
        for _ in range(N):
            state = _nl_body(state)

Requires: pure bodies, no bk.type inside the body, updates via bk.set_at, no
data-dependent python control flow in the body. Device residency is a
CONSEQUENCE of fusion, not a separate axis: inside a fused scan everything stays
in one XLA program.


4. THE FUSION-SCOPE SWITCH (the one gate that must survive)
-----------------------------------------------------------
The campaign is essentially finished; the winners are known. But ONE axis stays,
because physics will need it again:

  fuse across the time loop   |   fuse only up to the ns-loop

This is really "how much of the physics must live inside jax": whole-timeloop
fusion requires everything inside (incl. physics) to be jax-traceable; stopping
at the ns-loop lets non-traceable physics run per step outside the fused region.
The category boundary (section 13) IS this fusion-scope boundary.

Today this switch is NOT a real switch: the time-loop-fusion route
(run_timeloop_chunk / _step_core) hangs off the warm-up latch (section 6), so
selecting it is a wish whose activation point is decided at runtime by the
warm-up state machine. Moving the jit builds to setup (section 6/8) is what makes
it a genuine setup-time choice.

6. THE WARM-UP STATE MACHINE (the root knot) + the fix
------------------------------------------------------
Today three routes -- eager / per-nl jit / whole-loop scan -- and the code
TRANSITIONS BETWEEN THEM AT RUNTIME: a warm counter (_fuse_warm_calls), a steady
test (_nlbody_steady), a latch (_step_use_scan).

Why it exists -- the code says so: "Run eager until STEADY -- this builds the
sub-jits AND the _prepost_jit (else their BUILD would fire inside the
_nl_body/scan trace = illegal)." i.e. the sub-jits are built LAZILY, and the
BUILD site sits inside the nl-loop, so the loop must run eagerly to reach it.

Verified this session (pure JAX, on CPU -- no GPU needed; kept as the template
test test/prepost_jit_semantics_test.py):
  - jax.jit(fn) only WRAPS; it does not compile. Compilation fires on first call.
    "Build at setup" therefore needs no data and costs no compile.
  - Building a jit inside a trace is not itself illegal; USING that jit OUTSIDE
    the trace raises UnexpectedTracerError (a baked tracer leaks). That is the
    concrete meaning of the code's "illegal".
  - A jit that bakes a value on its first build and is reused on a later trace
    does NOT error when that value is stale -- it SILENTLY returns a wrong
    answer (shapes here are stable across steps, so nothing ever surfaces it).
    => purifying the body is the ONLY check on the "first closure reused
    forever" invariant, not cosmetics.

THE FIX: build the jits at setup (section 8). Feasibility CONFIRMED: _prepost_fn
closes over run-constants only (msc, bndc, GSGAM2, pre_bs, rho_bs, cnst scalars,
index consts -- the code's own comment certifies them "run-constant, set-once");
only 4 array args are traced. bk.device_consts already exists as the device-
resident run-constant mechanism. Build at setup => the warm counter, steady test,
latch and eager warm-up passes all disappear, and the route becomes a static
choice. Two lazily-built jits differ:
  _prepost_jit                 closes run-constants only -> buildable at setup TODAY
  _nl_body_jit / _nl_scan_jit  close over 17 per-step nonlocals -> need the body
                               purified first (section 7) before they can move.


7. THE STATE CONTAINER (msc): SPLIT THE BAG (immutable value / mutable registry)
--------------------------------------------------------------------------------
msc = StateContainer(), a Python-era invention (NOT from Fortran NICAM, which
uses module globals). It is a FLAT registry of ~30 subsystem singletons (bk,
cnst, adm, comm, grd, gmtr, oprt, vmtr, tim, rcnf, prgv, dyn, bndc, bsst, numf,
vi, src, srctr, trcadv, frc, io, ...), built and wired BEFORE the time loop.
Contract (driver-dc.py L66): "refer as msc.xxx after loading". load() = setattr.

The design INSTINCT -- "hold the model state as one object" -- is RIGHT and is
exactly JAX's own idiom (Flax/Equinox/Optax all thread a state object). What
fights JAX is not "object" but "a MUTABLE bag reached by reference and written
back". Resolve by splitting msc's two jobs, along the MUTABILITY axis:

  (A) SETTINGS/SERVICES bag -- run config, grid metrics, constants, the
      subsystem objects and their methods. Content does NOT change in the loop.
      MAY stay a plain mutable object: it never enters a jit trace, is read-only
      in the loop, so its mutability is harmless (used only to build it at
      setup). This is the msc we keep -- the single handle the design wanted.

  (B) PROGNOSTIC STATE value -- PROG/DIAG/PROGq. Content changes every step.
      Make it ONE IMMUTABLE VALUE (a pytree/dataclass) threaded through the pure
      body: new_state = step(state). "Immutable" = never edited in place; each
      step produces a NEW value and rebinds (like x=5 then x=6 -- 5 didn't
      become 6). Required because jax arrays are immutable AND because fusion/
      scan threads the carry as a value, not a mutated attribute.

"Unchanging settings may be mutable, changing state must be immutable" is not a
paradox: unchanging content never exercises the mutability (harmless); changing
content, to change under JAX, must be expressed as whole-value replacement.

Measured pull toward (B): the prognostic state is currently held FOUR ways --
as an attribute (self._prgvar_d), as a tuple carry (_prog_carry_d, _DIAG_carry,
...), as 17 nonlocals, and via msc (msc.dyn.PROG x6, msc.dyn.DIAG x9). Unifying
those into one immutable value IS the purification (section 8 stage 4) and is
MORE of the "state as an object" the design wanted, not less.

What msc does in the hot loop is NARROW (measured): msc.bk = 166 of ~250
one-level refs, ~111 of ~120 hot-loop refs, almost all msc.bk.resident (78) /
ndtype (29) / xp (17) / type (14) -- i.e. PROBING THE BACKEND for the route.
resident() is set-once (reads PYNICAM_RESIDENT once, caches, then returns
type=="jax" and const -- VERIFIED constant post-config), so these probes can be
resolved at setup and the ~111 hot-loop msc accesses removed with zero behaviour
change. That also makes maybe_jit's "one place" docstring true in the loop.

The consumer split (dynamics / physics / shared) is a LATER, orthogonal move
(it maps onto section 4's fusion-scope boundary) and is empty on the physics
side until physics lands -- so bundle by role first, by consumer later.


8. dynamics_setup + dynamics_setup_finalize
-------------------------------------------
dynamics_setup today (measured) does two small things and prepares NONE of the
loop's tools: (i) fills the RK large/small-step tables from TIME_integ_type, and
(ii) calls the sub-setups (BNDCND, bsstate, numfilter, vi). It is a NICAM-lineage
gather-and-dispatch method; NICAM users' mental map depends on that shape. It
already takes 16 EXPLICIT deps (comm, gtl, cnst, grd, gmtr, oprt, vmtr, tim,
rcnf, prgv, tdyn, bndc, bsst, numf, vi, rdtype) and NO msc -- honest injection,
the counter-example that makes "explicit args" viable.

DECISION: keep dynamics_setup's structure unchanged. Add the Python/JAX-era
wiring as a separately named dynamics_setup_finalize(), called at the END of
dynamics_setup (after the sub-setups, so bsst.rho_bs/pre_bs are already built --
a natural ordering guarantee). Rationale:
  - NICAM mental map intact: dynamics_setup still reads as the classic setup.
  - The acceleration wiring is quarantined behind a name that says so; its
    docstring states exactly what it does (resolve route, bake run-consts, build
    jits) to back the generic name with a contract.
  - It maps section 2's structure/no-structure axis onto a method boundary.
  - Works on both backends: maybe_jit is a no-op on numpy, device_consts trivial,
    so finalize just "does less" on numpy; the reference path stays clean.
Driver keeps calling dynamics_setup only -- its mental map does not move either.

dynamics_setup_finalize's job (filled incrementally by the work order):
  - resolve the route once (self._resident/_xp/... from bk) -- section 7
  - bake run-constants via bk.device_consts -- section 6
  - build _prepost_jit / _prepost_pl_jit, then (after purification)
    _nl_body_jit / _nl_scan_jit / _step_core -- section 6

10. MEASUREMENT
---------------
Current practice (discard the first ~2 compile+warm-up steps, measure steady
sec/step) stays valid, but today's design ENTANGLES compile with warm-up: the
compile point is set by the warm-up machine and MOVES with RK order (2/3/4), so
the discard window can only be guessed.

  after section 6/8   compile is confined to the first call -> clean boundary
  with AOT            jax.jit(f).lower(ShapeDtypeStruct...).compile() at setup
                      -> compile becomes a NAMED, directly measured setup phase;
                         shapes are static after setup, so no real data needed.
  regression test     JAX_LOG_COMPILES=1; assert "zero compiles in the
                      measurement window" -- also catches retraces from set_at or
                      new physics later. (Note: jax.jit WRAPS; moving
                      CONSTRUCTION to setup does not by itself move COMPILATION.)

13. PHYSICS CATEGORIES AND FILE LAYOUT
--------------------------------------
Physics enters through two structurally different doors:
  DC style     forcing is ADDED TO f_TEND inside the dynamics step
               (mod_forcing_driver, one thin AF_TYPE hook)
  NICAM style  physics is OPERATOR-SPLIT: physics_step modifies the prognostic
               variables between dynamics steps (CP->SC->MP->RD->SFC/TB, fixed
               order, sub-stepping), using prgvar / diagvar / sfcvar

Full NICAM files artificial forcing UNDER physics/; "forcing vs physics" is a
DC-ism. Prefixes encode SCOPE, not lineage: af_ = a complete artificial forcing
(both trees), fr_ = a friction term only (nicam, via FR_TYPE). Mirroring both
trees verbatim would leave forcing-like code in two directories.

DECISION: keep the name nhm/forcing/, but make it a RULE about content:
   nhm/forcing/      ALL artificial/idealized forcing, any lineage
                     (today: af_dcmip, af_heldsuarez, af_trcadv, mod_forcing)
   nhm/share/dcmip/  self-contained idealized TEST PHYSICS + initial conditions
                     (kessler, simple_physics, terminator, dcmip_ic)
   nhm/physics/      real parameterizations only: CP / MP / RD / TB / SF
   One concept -> one place. Cost: a documented deviation from nicam for its
   forcing modules, largely notional (DC's mod_af_heldsuarez is a verified
   SUPERSET of nicam's; nicam's mod_fr_heldsuarez is a friction-only fragment,
   redundant once the complete af_ version is present).
   Rejected: dc_forcing/ (still two places; hyphen illegal in a package name);
   move forcing/ into share/dcmip/ (mislabels Held-Suarez, which is H&S 1994).

DRIFT TO FIX: kessler.py, simple_physics.py, terminator.py sit in nhm/forcing/
but nicamdc keeps them in share/dcmip/ (mod_dcmip_ic.py is already in nhm/share/).
Move them to nhm/share/dcmip/ -- realigns with Fortran AND implements the rule.

The physics scheme STYLE is already correct and is the template: kessler.py and
simple_physics.py are module-level, xp-parametrized (numpy-first), column-
vectorized functions returning new arrays. kessler's rain sedimentation has a
DATA-DEPENDENT trip count (illegal under jit) and solves it exactly as section 3
prescribes: shared body (_rain_body), branch only at the loop driver (python for
| lax.fori_loop). mod_dynamics is the deviant, not the physics.
  Known gap: simple_physics TC_PBL_mod (Bryan PBL) is not xp-wired (downward
  recurrence, raises on jax). Same fix: shared body, branch at the loop driver.
Open: mod_nudge is filed under forcing/ by nicamdc and share/ by nicam -- pick one.


14. NAMING: self. / _x / __x   (settled)
----------------------------------------
Measured over 75 classes / 1075 self attributes: plain 842 (78%), _x 218 (20%),
__x 0. DECISION:
  [+] self.x   real state: survives across calls, or read by another method/object
  [+] local    scratch. DEFAULT TO LOCAL; self. needs a reason
  [+] _x       internal: caches, jits, bookkeeping -- not the module's contract
  [-] __x      DO NOT USE. Name mangling solves subclass collision; these are
               module-bottom singletons with no subclasses -> solves a non-problem.
DEBT: 361 attributes written+read in ONE method only -- "should have been a local"
candidates (UPPER BOUND: some are legit cross-call jit caches that MUST live on
self). Triage: does the value survive between calls? yes -> self._x  no -> local.
